# 移车出库（Rush Hour / 停车场迷宫）开发经验归纳

> 本文是一份「交接说明」，可直接作为上下文交给另一个 agent，用于从零开发一款同类滑块解谜游戏。
> 按「游戏怎么搭 + 关卡怎么录入 + 踩过的坑」三块组织。

## 1. 目标与技术栈

- 纯 **HTML + CSS + 原生 JavaScript**，无框架、无构建步骤，方便直接部署到 GitHub Pages（作为个人主页 `用户名.github.io` 的根目录）。
- 6×6 固定棋盘。车辆只能沿自身朝向（水平/垂直）滑动，不能转向、不能跨越其他车。
- 目标车（红/黑）开到出口即胜利。
- 功能：移动（拖拽 + 触屏）、步数、计时、重置、撤销、关卡选择/上一关/下一关、胜利动画、最佳成绩（localStorage）、移动端适配。

## 2. 核心数据模型（最重要的一块，先定好它）

所有关卡是一个数组，每个关卡形如：

```javascript
{
  id: "p001",                        // 关卡稳定唯一标识，一经分配【永不更改、不复用】
  name: "入门",                       // 可选；缺省时显示为「第 N 关」
  optimal: 9,                        // 占位即可，加载时被 BFS 算出的真实最优覆盖
  stars: 3,                          // 可选；手动指定 1~6 星难度，优先于自动分级
  exit: { side: "bottom", lane: 3 }, // 可选；缺省 = 右侧第 3 行（经典出口）
  cars: [
    { id: "X", x: 3, y: 0, len: 2, dir: "v" }, // X = 目标车
    { id: "A", x: 0, y: 1, len: 2, dir: "h" },
    // ...
  ],
}
```

约定：

- 坐标 `x`（列）、`y`（行）**从 0 开始**，指车头（最左/最上格）。
- `len` ∈ {2, 3}，`dir` ∈ {"h", "v"}。
- 车辆里的 `id: "X"` 固定是目标车。
- **从一开始就把 `exit` 设计成「四方向通用」**，不要写死右侧出口。我们中途才发现图片题库的出口在底部，被迫返工改引擎。`exit.side` ∈ {right, left, top, bottom}，`exit.lane` 是出口所在的行或列索引。

### 2.1 命名 / 编号 / 难度（为数百关设计，务必照做）

把**三件事彻底解耦**，否则一旦到几十上百关，重排/插队就要全员改名（血泪教训）：

| 概念 | 来源 | 是否随顺序变化 |
| --- | --- | --- |
| **关卡 `id`** | 手填稳定值 `p001…`，永不更改 | 否（身份标识） |
| **显示编号** | 「第 N 关」由**数组下标实时计算** | 是（自动重编号） |
| **显示难度** | 6 级星标 ★，由 `stars`（手填优先）或 `optimal` 步数自动换算 | 跟谜题本身走 |

- **最佳成绩用 `id` 作 localStorage 的 key**（`STORE_KEY` + `{ [id]: 步数 }`）。这样随意重排、在中间插入新关，历史成绩都不会错位/丢失。
- **6 级星标自动分级**（`game.js` 里 `STAR_TIERS = [3,6,9,14,22]`）：最优步数 ≤3→1★, ≤6→2★, ≤9→3★, ≤14→4★, ≤22→5★, 其余→6★。需要时给关卡加 `stars` 字段手动覆盖。
- **导入新题 = 往数组里追加 `{ id: "pNNN", cars, exit }`**（`id` 取下一个未用过的编号，与放在第几位无关）。编号、星级、下拉框文案全自动生成，**不需要手动改名**。
- 重排/移动某关只是改数组位置（或加一个 `order` 字段排序），显示编号会自动更新。

## 3. 引擎要点

- **广义出口**：胜利判定、出口标记渲染、逃逸动画都要根据 `exit.side` 动态计算，不要假设方向。`getExit(level)` 返回该关出口（缺省回退到经典右侧）。
- **胜利判定**：目标车贴着出口那一侧、且朝向与出口一致、且出口外侧（朝出口方向）那条通道全空，即视为可出库。
- **渲染**：用 CSS Grid / 绝对定位按 `x, y, len, dir` 摆放；出口标记用一组 `exit-right/left/top/bottom` class 控制位置。
- **动画**：胜利时让目标车沿出口方向平移出界，方向由 `exit.side` 决定（别写死 `left: 120%` 之类）。

## 4. 内置 BFS 求解器（强烈建议，省一半事）

在前端内置一个 BFS 求解器 `solveOptimal(cars, exit, maxStates)`：

- 启动时异步（`setTimeout` 分片，避免卡 UI）跑一遍所有关卡：
  - 无解 → `console.error` 报警（开发期立刻发现坏关卡）；
  - 有解 → 用**真实最优步数覆盖手填的 `optimal`**；
  - 车太多 / 状态爆炸（如 >10 辆或超过 `maxStates`）→ 跳过，保留手填值。
- 价值：**只要录入的车位是对的，求解器自动判可解性 + 算最优步数**，不需要人工去解每一关。这点对「图片录入」工作流是决定性的。

## 5. 从图片录入关卡的工作流（核心经验）

这是耗时最多、最容易错的环节。固化成下面的流程最稳：

1. **先要高清图**。低分辨率会把朝向/长度/列位看错，直接让用户重发清晰图。
2. **画 ASCII 网格**：用 `列1..列6 / 行1..行6` 的表格把每辆车摆出来，**逐车标注把握程度**（"较有把握 / ❓ 不确定"）。
3. **主动列出疑点并提问**，尤其：目标车占哪两行/列、长车是 len2 还是 len3、相邻同色块到底是一辆长车还是两辆、出口那一列下方有没有被挡。
4. **用户确认/纠正后，再写入数据**。不要凭猜测直接写。
5. **写入前必须验证可解性**：要么内置 BFS 自动判，要么让用户**直接发通关步骤**，照着走一遍。通关步骤是最可靠的「对答案」方式——既验证可解、又反推出正确车位。
6. 写入 `levels.js`，`optimal` 填占位，交给 BFS 覆盖。

## 6. 易错点与教训（务必转告新 agent）

- **ASCII 画图常漏格**：竖向长车经常只画了头格、漏了尾格（"紫色车应在行 2–3，你只画了行 2"）。画完自己对一遍每辆车的 `len`。
- **「看似无解，其实可解」是最大陷阱**。多次凭手工推演断定「死锁/无解」，结果用户发来通关步骤证明可解。原因几乎都是漏掉了这两类技巧：
  - **临时腾挪窗口**：让目标车/某辆挡路车先「上移再下移」（或某长车「左移再右移」归位），在这个**短暂空档**里让第三辆车一口气横穿过去（例如黄车 `右移 4` 直接穿到最右），事后再各自归位。
  - **连锁释放**：清掉底部长卡车后，它原先占的格子空出来，往往**解锁了另一条竖车下移/某车右移**的通道，从而盘活全局。
  - 教训：**不要轻易下「无解」结论**。先穷举更深几层，或直接请用户给步骤；手工 BFS 不可靠，能上程序就上程序。
- **出口方向**：图片题库目标常是「竖向车 + 底部出口」，与经典「横向车 + 右侧出口」不同。引擎一定要通用化。
- **同色相邻块的歧义**：一辆「红头灰身 / 绿头棕身」的长车，很容易被看成两辆。务必跟用户确认「是一辆还是两辆」。

## 7. 部署（GitHub Pages）

- 直接把 `index.html`、`css/`、`js/` 放到 `用户名.github.io` 仓库**根目录**，`git push` 后 1–2 分钟自动构建生效。
- 沙箱环境里 agent 常常**无法直接执行 git/node**；这种情况下让用户在本地 PowerShell 跑 `git add/commit/push`，并把本地 BFS 验证脚本（可选的 `validate.mjs`）也交给用户在本机跑。

## 8. 推荐文件结构

```
项目根/
├── index.html          # HUD + 棋盘 + 控制区 + 胜利遮罩
├── css/
│   └── style.css        # 车辆、棋盘、出口标记（四方向）、胜利动画
├── js/
│   ├── levels.js        # 关卡数据（见 §2）
│   └── game.js          # 引擎：BFS 求解器、移动、撤销、计时、渲染、胜利判定
├── validate.mjs         # （可选）本地 Node 版 BFS 校验脚本
├── docs/
│   └── GAME_DESIGN.md   # 本文
└── README.md
```
